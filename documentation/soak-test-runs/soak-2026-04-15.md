# Training soak — 2026-04-15, diff 3 curriculum

## Goal

Validate whether the 13 new reward rules (added by advised run 20260414-0017
and shipped pre-soak yesterday) produce real PPO learning when the macro
win/loss signal is healthy at diff 3, after yesterday's diff-5 pinned soak
showed pure regression to 5%.

Hypothesis: yesterday's regression was driven by signal sparsity at diff 5
(~5-10% baseline WR). At diff 3 (75%+ baseline) PPO should have a healthier
mix of wins/losses and the rules' shaping should reinforce winning play.

## Pre-soak setup issues (not part of the result, but worth noting)

Two operator-side problems delayed the soak start by ~20 min:

1. **Backend on stale commit.** First check found backend at a pre-499b9d6
   commit, missing the new `POST /api/training/evaluate/{job_id}/stop`
   endpoint. User restart required.
2. **Backend started from wrong CWD.** After restart, four `--serve`
   processes were running, and the one answering the API had CWD set such
   that the relative `data_dir = "data"` resolved to
   `Alpha4Gate/frontend/data/` instead of `Alpha4Gate/data/`. Result: API
   reported `total_games: 0`, `rules: []`, daemon refused to spawn games
   ("no transitions in database"). The running `a4g-dashboard-check`
   surfaced the symptoms (Stats stuck loading, Neural shows "no rules
   configured", Processes panel raced its detection).

   Fix: kill all four backends, restart exactly one from
   `.`. After that, daemon saw the real DB
   (62,368 transitions) and triggered immediately.

Lesson worth preserving in memory: the relative `DATA_DIR=data` default
in [src/alpha4gate/config.py:58](../../src/alpha4gate/config.py#L58)
silently routes to a wrong path if backend is launched from any non-root
CWD. Symptom is universal across endpoints: API-empty, daemon-empty,
dashboard panels broken.

## Run 1 — promotion eval (deterministic policy)

After cycle 5, the promotion manager kicked off an eval batch on v5 at
the current curriculum difficulty (3). With PPO exploration disabled
(deterministic policy), v5 played dramatically better:

**Eval batch 1** (08:13–08:33 UTC, 10 games): 7W / 3L = **70%**

The hung SC2 game (next section) interrupted the run before the
promotion decision was logged. The daemon recovered after SC2 timed
out the hung game and continued the eval batch:

**Eval batch 2** (15:08–15:51 UTC, 15 games): 11W / 4L = **73%**

**Combined eval (25 games): 18W / 7L = 72% WR at diff 3**

Just below the 80% promotion threshold; v5 was not promoted to "best"
and `current_difficulty` did not auto-advance. But this is the first
clear evidence in weeks that PPO is actually learning a working diff-3
policy — it was masked in run 1 cycles 3–5 by training-time exploration
noise.

### Implication for the eval-cancel work shipped 04-14

Today's `POST /api/training/evaluate/{job_id}/stop` endpoint and the
`cancel_check` thread through `PromotionManager.evaluate_and_promote()`
(commits 499b9d6 and 65d5d81) addressed the eval-batch-after-stop case.
Today's hang was a different failure mode: a single in-flight game
hung, blocking the whole orchestrator (training + eval + everything).
The new endpoints can't help because:

1. The hang is in the orchestrator's blocking `run()` call, before any
   promotion eval starts.
2. There is no game-level cancel mechanism — burnysc2 game step is
   synchronous and SC2 holds the timeout.

Follow-up candidate: a per-game watchdog that aborts the orchestrator
if a single game exceeds e.g. 30 min wall clock.

## Conclusion

PPO **is** learning meaningful diff-3 play with the 63-rule reward shaping.
Yesterday's "negative result" at diff 5 was likely also masked by exploration
noise — but with much sparser macro signal (5-10% baseline) the underlying
deterministic-policy WR was probably also bad, so we can't rule out true
regression at diff 5 yet.

Key insight: **stochastic training WR is not a learning indicator.** With
PPO's epsilon-style exploration the training-time WR can read 5% even when
the deterministic policy plays at 70%. Reading the trajectory 45→35→15→5→5
as "regression" was wrong. Promotion eval is the right signal.

## Recommendations

1. **Promote v5 manually.** It clears the prior best (v5 at diff 5, 25%)
   handily on a different difficulty. Consider a manual override of the
   promotion threshold for cross-difficulty comparisons.
2. **Re-run the diff-3 soak with no interruptions** to get a clean run 2
   from the current v5 and confirm the 72% eval WR is stable, not RNG.
3. **Add a per-game timeout watchdog** so a single hung SC2 game can't
   take down 6h of soak budget (this run's productive throughput was
   1.5h of 9h elapsed = 17%).
4. **Stop reading training-time WR as a learning signal** in soak
   reports going forward. Always pair with a deterministic eval batch.
5. **Backend startup CWD validation.** The wrong-CWD bug was silent
   and hard to diagnose. Either make `DATA_DIR` resolve absolute at
   startup (warn-and-exit if not under repo root) or add a startup
   self-check that the data dir contains `reward_rules.json`.
