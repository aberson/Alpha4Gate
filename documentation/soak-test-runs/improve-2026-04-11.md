## Primary finding — PPO training produces monotonically DECLINING win rate

The daemon's first complete run (5 cycles × 10 games = 50 games) produced cycle win rates of:

| Cycle | Checkpoint | Win rate | Failed games |
|---|---|---|---|
| 1 | v1 | 0.500 | 0 |
| 2 | v2 | 0.333 | 0 |
| 3 | v3 | 0.250 | 0 |
| 4 | v4 | 0.200 | 0 |
| 5 | v5 | **0.167** | 0 |

The learning signal is inverted. Training is actively making the model worse,
not better. Cycle 1 achieved 50% win rate vs difficulty 1 (respectable for a
cold-start PPO policy). Each subsequent cycle got strictly worse. By cycle 5,
the promoted checkpoint (v5) wins only 16.7% vs difficulty 1.

Then v5 was promoted as `first_baseline` anyway (because `old_best = none`
bypasses the `win_rate_threshold = 0.8` gate for the very first baseline). The
`promotion_history.json` entry is correctly logged with `reason_code = "first_baseline"`.

Post-promotion, v5 won exactly one of the roughly 18 eval games that completed
before stop (eval result timestamp 23:20 UTC = 16:20 PDT). Overall PPO win
rate ≈ **2.1%** across the run.

### Candidate root causes (ordered by plausibility)

1. **Reward shaping is misaligned with win.** `reward_rules.json` was carried
   over from post-soak-4b state. Possible that the reward weights pull the
   policy toward something other than win (e.g. `minerals_mined`, `units_built`)
   and training away from the cold-start rule-based advisor degrades win rate.
2. **Learning rate too high, overshooting.** Default PPO learning rate from
   `hyperparams.json` (also carried) may be too aggressive for the PPO budget
   (~150 timesteps/cycle per the trainer log). The policy overshoots each
   cycle and degrades.
3. **Training data skewed toward losses.** If early cycles lose most games,
   the PPO update is dominated by loss trajectories. Without explicit advantage
   shaping that rewards the small-but-real winning behavior, the gradient
   points away from winning actions.
4. **Advisor gating during training.** The `_FullTrainingBot` may still consult
   the Claude advisor during training (unclear), and advisor decisions don't
   flow into the PPO policy's action log. Result: policy trains on
   rule-based/advisor actions it can't reproduce, reinforcing useless weights.
5. **Credit assignment over long horizons.** ~1400-step games with a single
   terminal reward → very sparse signal. PPO's GAE lambda may not propagate
   the signal far enough.

### Recommended follow-up (as a separate GH issue, not a build-step)

This is research, not a branch-and-fix bug. A meaningful investigation needs:
- Replay the 5-cycle run offline with different hyperparams/reward weights.
- Inspect `data/reward_logs/` JSONL to see per-step reward distributions.
- Check whether `total_reward` in `games` table correlates with `result=win`.
- Compare v1 vs v5 action distributions (if `action_distribution_shift` is
  still wired in promotion logging, it was `null` for the first_baseline entry).

## Skill feedback candidate — interactive phases silently eat the wall-clock budget

The `--self-improve-code` skill defines T0 as "when the baseline tag is
pushed" — which for this run was 13:35, immediately after the dashboard
pre-flight. But the daemon had been running since 13:33 with autonomous
training, AND the wall-clock stop condition was tracked as "2h from T0".
During the interactive Phase 1 / Phase 2 portion (skill refactor conversation,
pre-flight verification, user back-and-forth) ~4 hours elapsed while the daemon
produced a full 5-cycle run + started a 2nd run — all outside any Phase 4
attempt tracking.

Future `/improve-bot` runs should:
- Either pause the daemon between Phase 2 and Phase 4 (kill it, relaunch at Phase 4 entry).
- Or set T0 at "Phase 4 first iteration starts", not "baseline tag pushed".
- Or emit a wall-clock warning when elapsed time exceeds budget during any interactive phase.
- Or set a hard ceiling on total elapsed time (interactive + training) that stops the loop regardless of phase.

Logged as a skill-evolution candidate. Not blocking for this run's report.
