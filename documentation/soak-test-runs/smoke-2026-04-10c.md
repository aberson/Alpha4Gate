## Smoke 3a — F6 surfaced

| t | event |
|---|---|
| t≈11:53 | backend started |
| t+6s | `/api/training/daemon` → `running:true, state:idle, runs_completed:0, current_difficulty:1, cycles_per_run:1, games_per_cycle:1` ✓ |
| t+76s | first check tick → `state:training` briefly, then back to `idle` in milliseconds |
| t+76s | `last_result.cycle_results[0]` → `{cycle:1, difficulty:1, status:"crashed", error:"ValueError: Action spaces do not match: Discrete(5) != Discrete(6)"}` |
| t+~80s | operator killed backend cleanly via `cmd.exe //c "taskkill /T /F /PID <pid>"` |

**Diagnosis:** [`trainer.py:_init_or_resume_model`](Alpha4Gate/src/alpha4gate/learning/trainer.py#L311) hardcoded `act_space = spaces.Discrete(5)` and only the obs space read from `FEATURE_DIM`. Meanwhile [`environment.py:SC2Env`](Alpha4Gate/src/alpha4gate/learning/environment.py#L57) defines `action_space: spaces.Discrete = spaces.Discrete(6)`. Same anti-pattern as F1 (silent space drift between model init and env), different dimension. The fresh model was built with 5 actions, the env has 6, set_env raises.

**This is finding F6 — sibling of F1.**

## F1+F6 fix applied between smoke 3a and 3b

`_init_or_resume_model` no longer hardcodes spaces. It imports `SC2Env` and reads `SC2Env.observation_space` and `SC2Env.action_space` directly. Single source of truth: the env producer. The model can never silently drift from the env again.

```python
# OLD
obs_space = spaces.Box(low=0.0, high=1.0, shape=(FEATURE_DIM,), dtype=np.float32)
act_space: spaces.Discrete = spaces.Discrete(5)

# NEW
from alpha4gate.learning.environment import SC2Env
dummy_env.observation_space = SC2Env.observation_space
dummy_env.action_space = SC2Env.action_space
```

**Regression test added:** `TestModelInitMatchesEnv::test_fresh_model_uses_sc2env_spaces` asserts `model.observation_space == SC2Env.observation_space and model.action_space == SC2Env.action_space`. Will catch any future drift.

Gates after fix: 687 pytest (was 686), 0 mypy, 0 ruff.

Different surface (DB schema instead of gym spaces) but identical root cause shape.

**This is finding F7.**

## Pattern observation: same anti-pattern, three instances in one session

| # | Symptom | Producer | Consumer | Drift |
|---|---|---|---|---|
| F1 | obs space mismatch | env (`SC2Env.observation_space`) | model init | 15 → 17 dims |
| F6 | action space mismatch | env (`SC2Env.action_space`) | model init | 5 → 6 actions |
| F7 | sqlite OperationalError | bot transition recorder | DB schema | added `cannon_count` field |

**All three are the same shape:** a producer in the loop's data pipeline added or changed a field, and at least one consumer in the same pipeline didn't migrate. The 682-test suite caught zero of them because each test mocks the boundary it would have asserted on.

**This is now a process question, not a coding question.** Three instances in one debugging session strongly suggests there are MORE waiting in the pipeline. The right move is **NOT** to keep playing whack-a-mole one smoke test at a time. The right move is to **audit the whole pipeline for this pattern in one pass**:

1. `git log --oneline -- src/alpha4gate/learning/ src/alpha4gate/bot.py` since whatever commit introduced the 17-dim obs space.
2. For every commit that added a field/dimension/column, check whether the matching consumer was updated in the same commit.
3. Likely candidates to grep: any `record_transition`, `store_*`, observation building, action enumeration, reward calculation. Plus any schema/migration files for the SQLite DB.
4. Build a single comprehensive fix PR rather than smoke-test-then-fix loops.

Until that audit happens, smoke 4 will likely surface F8, smoke 5 will likely surface F9, etc.
