## User question answered: "is the daemon-training bot different from the runtime bot?"

Yes, intentionally. Both are `Alpha4GateBot` ([bot.py](src/alpha4gate/bot.py)) with identical macro/micro/scouting code. The difference is **how the strategic state is chosen**:

- **Runtime (hybrid/neural inference):** PPO picks the mean/greedy action every time (exploit).
- **Daemon training:** PPO *samples* from its policy distribution (explore). Entropy coefficient makes it deliberately pick suboptimal states to learn their value. Behavior looks chaotic because it is — by design.

See [environment.py:684-700](src/alpha4gate/learning/environment.py#L684) for the `_FullTrainingBot` that wires PPO into `Alpha4GateBot` via a `_GymStateProxy`.

## User question answered: "can the daemon run headless / batch games?"

**Headless:** No — SC2 always renders. But you can minimize the window via burnysc2 `proc_args` (`--windowmode 1 --windowwidth 320 --windowheight 240`) so rendering cost is negligible.

**Batch/parallel games:** Yes, but needs code. burnysc2 supports `Portconfig` to run multiple SC2 instances on distinct ports — you'd orchestrate N `run_game` calls concurrently. 4 parallel games ≈ quarters the soak time. Each SC2 instance uses ~500MB-1GB RAM.

This would be a great `--self-improve-code` target — add a `ParallelGameRunner` to `src/alpha4gate/connection.py`, wire it into the daemon's rollout phase, drop daemon soak from 30 min → 8 min.
