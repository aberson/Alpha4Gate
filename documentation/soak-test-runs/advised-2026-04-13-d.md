**Phase 5 — Validation (3 games, diff 5, max speed):**

| Game | Result | Duration | Reward |
|------|--------|----------|--------|
| 1e1ef5f74baf | LOSS | 12:44 | -111.2 |

**Validation batch: 1W-2L (33%).**

**Delta: 67% → 33% = -34% absolute.** Exceeds the 30% fail threshold.

**Result: MARGINAL (keeping change).** Reasoning:

1. **Rules-mode doesn't exercise PPO.** The new reward rules influence PPO training signals only. The bot runs `--decision-mode rules` by default — so the validation games ran identical behavior code to the observation games. Validation variance is bot/SC2 randomness, not a signal about the rule change.
2. **n=3 variance.** A single-game swing at n=3 is 33%. 2W-1L vs 1W-2L differs by one game. This is below the noise floor for a 3-game sample.
3. **Additive, asymmetric penalty.** The rules only fire when the bot is supply-capped late-game — the exact failure pattern we saw. They never reward bad behavior; they never penalize good behavior.

Recommendation: next run at `--decision-mode hybrid` with 8-10 games/cycle to actually test PPO response.

## Stop Decision (after iteration 1)

**Stopping at 1 iteration despite ~41 min budget remaining.**

Reasoning: iteration 1 surfaced a methodological gap that makes additional iterations low-value:

- `--mode training` only modifies reward rules / hyperparameters, both of which influence **PPO training signals**, not bot gameplay.
- Default `--decision-mode rules` means PPO is **not consulted during play**. The bot runs identical rule-based decisions regardless of reward-rule edits.
- Therefore validation games in `--mode training` cannot distinguish rule-change signal from bot/SC2 variance. At n=3, one-game swings (±33%) exceed the default 30% fail threshold.
- A second reward-rule iteration would hit the same wall.

The productive closure: commit the reward rule additions (they'll influence PPO whenever the daemon runs next), document the methodology gap, and stop.
