## Pattern lesson learned

The Phase 4.5 audit found **four instances of the same anti-pattern**: a constant defined in two places, with the two copies drifting out of sync. F1 (obs space hardcoded in trainer vs FEATURE_DIM), F6 (action space hardcoded in trainer vs SC2Env), F8 (same in imitation), F9 (action list hardcoded in neural_engine vs environment). Each instance shipped because the duplicate copy looked self-contained and the unit tests mocked the boundary that would have caught the drift.

**Generalizable rule**: any constant that defines the shape of data (dimensions, column counts, action counts, schema) must have **exactly one source of truth**, and consumers must import it rather than redefining it. Tests should assert object identity (`is`), not just value equality, so a future regression that re-introduces a duplicate copy fails immediately even if the values happen to match at first.

This pattern is now baked into the regression tests in `test_decision_engine.py::TestActionToStateSourceOfTruth` and `test_database.py::TestLegacySchemaMigration`. Future drift in either direction (action list or DB schema) cannot ship past CI.
