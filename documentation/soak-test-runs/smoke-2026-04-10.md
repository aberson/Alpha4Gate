## What the wiring proved

These outcomes are **wins** — the daemon mode is wired correctly and behaves as designed:

1. **CLI flag is wired.** `--serve --daemon` starts the API server AND auto-starts the `TrainingDaemon` thread (`runner.py:184-189`).
2. **Daemon thread runs.** `GET /api/training/daemon` returns `running:true` immediately after server startup.
3. **State machine works.** Daemon advances `idle → checking → training` on the first `check_interval_seconds=60` tick.
4. **Trigger evaluation fires.** With 3203 existing transitions in `data/training.db` and `min_transitions=500`, the transition-count trigger fired on the very first check.
5. **Daemon-to-trainer plumbing works.** Daemon successfully constructed a `TrainingOrchestrator` and called `orchestrator.run(n_cycles=5, games_per_cycle=10)`.
6. **Promotion plumbing works.** After "completing" 5 cycles, the daemon ran `PromotionManager.evaluate_and_promote(v5, difficulty=6)`, which constructed a `ModelEvaluator` and started a 20-game evaluation pass against real SC2.
7. **SC2 integration works.** The evaluator successfully launched real SC2 games with `Map: Simple64, Computer VeryHard` and got to `Status.in_game`.
8. **API endpoints respond.** `/api/training/daemon` and `/api/training/status` returned valid JSON throughout the run.
9. **Process tree kill works** when invoked via `cmd.exe //c "taskkill /T /F /PID <pid>"` to bypass Git Bash's `/T` → `T:/` path translation. All 3 SC2 child processes were killed alongside the parent. User's manually-launched SC2 instance was untouched.

**Step 2 wiring goal: ACHIEVED.** The daemon mode is functional in the sense the spec asked about.

## Findings (3 BLOCKERS, 2 MAJORS)

### Finding 1 — BLOCKER — Observation space mismatch crashes every training cycle
**Category:** training
**Evidence:** `backend.log` lines around `Cycle 1 ... crashed` (and identical for cycles 2-5)

```
ValueError: Observation spaces do not match:
  Box(0.0, 1.0, (15,), float32) != Box(0.0, 1.0, (17,), float32)
  at trainer.py:158 (model.set_env)
```

The existing checkpoints in `data/checkpoints/` (best=`v1`) were trained with a 15-dimensional observation space. The current `_make_env(db)` builds an environment with a 17-dimensional observation space. SB3's `model.set_env` validates observation/action space compatibility and raises `ValueError`.

**Impact:** every training cycle in this smoke test failed at `set_env` BEFORE `model.learn()` could run. **Zero actual training happened.** The daemon ran 5 cycles in milliseconds because each one crashed instantly. New checkpoints `v1`-`v5` were saved but they are byte-for-byte copies of the unchanged in-memory model — no learning occurred.

**Root cause:** somewhere between when the v1 checkpoint was saved and now, the env's observation space gained 2 dimensions. Likely a feature added to `_make_env` or the bot's state vector that didn't get a checkpoint migration. The exact diff that introduced the new 2 dimensions needs to be hunted down (`git log --oneline -- src/alpha4gate/learning/` and grep for observation_space changes).

**Fix path (NOT done in this step — evidence collection only):** either (a) re-train from scratch and discard all v1-v13 checkpoints, or (b) migrate the old checkpoints by extending their observation space with zeros for the new dimensions, or (c) detect the mismatch on `set_env` and re-init the model from scratch instead of crashing.

### Finding 2 — BLOCKER — Trainer reports crashed cycles as successful
**Category:** training
**Evidence:** `trainer.py:157-209`, backend.log "Training cycle N crashed" followed by "Cycle N complete: ..." for each cycle

`TrainingOrchestrator.run()` wraps `model.set_env(env)` and `model.learn(...)` in a try/except (trainer.py:157-172). On exception it logs "Training cycle N crashed", increments `total_games += 1`, and **falls through to the success-path code below the try block** (lines 174-209):

- Reads `recent_win_rate = db.get_recent_win_rate(games_per_cycle * 2)` — returns stale data from prior runs (in this smoke test: `0.875` from the 8 games already in the DB)
- Calls `should_increase_difficulty(recent_win_rate)` and may auto-advance the curriculum
- Logs `"Cycle N: win_rate=0.88"` as if this cycle had earned that win rate
- Calls `save_checkpoint(model, ..., is_best=False)` — saves a fresh checkpoint named `v{cycle}` even though the model is byte-for-byte unchanged from the cycle start
- Appends `cycle_result = {cycle, difficulty, win_rate, checkpoint}` to `results` and logs "Cycle N complete: ..."

**Impact:** the daemon's `_run_training()` then receives a `result` dict that says all 5 cycles completed with `win_rate=0.875`, persists `final_difficulty=6` to `daemon_config.json`, and runs the promotion gate on the unchanged-but-renamed `v5` checkpoint. Every observable signal — `runs_completed` counter, dashboard "successful cycles", curriculum advancement — is silently lying.

**This is the most dangerous bug in the loop.** A long unattended run would advance the curriculum to max difficulty, accumulate dozens of fake "successful" cycles, and the operator would have no signal that anything was wrong unless they grep the log for "crashed".

**Fix path:** the try/except should either re-raise after logging, OR set a `cycle_failed = True` flag and skip the entire post-training block (no win_rate read, no curriculum advancement, no checkpoint save, no result append). The cycle_result should be `{"cycle": N, "status": "crashed", "error": str(exc)}` and the daemon's `_run_training` should detect crashed cycles and bail out instead of running the promotion gate on a phantom checkpoint.

### Finding 3 — BLOCKER — Curriculum auto-advances on phantom training
**Category:** training / daemon
**Evidence:** backend.log "Difficulty increased to 2" through "Difficulty increased to 6" across the 5 crashed cycles; final `data/daemon_config.json` (now persisted) has `current_difficulty: 6`

Direct consequence of Finding 2. `should_increase_difficulty(recent_win_rate=0.875)` returns true because 0.875 > `win_rate_threshold=0.8`, and since the trainer reads stale-DB win rates after each "completed" cycle, the curriculum advances on every cycle. In one 60-second smoke test, the curriculum jumped from difficulty 1 to 6.

**Impact:** even after Finding 2 is fixed, the curriculum logic should not blindly advance on win rates pulled from a global DB query — it should only count games played within the current cycle. Otherwise the very first cycle (which inherits the previous run's win rate) will always advance.

**Fix path:** `db.get_recent_win_rate(N)` needs to be cycle-scoped, OR `should_increase_difficulty` needs to also check that at least N new games were played this cycle. Compare `total_games` before/after.

**Side effect to clean up:** `data/daemon_config.json` was written by this smoke test with `current_difficulty: 6`. This file did NOT exist before the test. The operator may want to delete it before the real Step 3 soak run.

### Finding 4 — MAJOR — Eval games launched real SC2 from a smoke test
**Category:** daemon / safety
**Evidence:** backend.log shows `sc2.controller create_game` and `Status.in_game` after the smoke test "completed" 5 cycles in milliseconds

The daemon's `_run_training()` runs the promotion gate (`PromotionManager.evaluate_and_promote`) on the latest checkpoint immediately after a training run completes. The evaluator launches real SC2 games. So a smoke test of "verify the daemon starts" cascaded into actually launching SC2 games against the user's main-menu SC2 instance.

**Impact:** any future operator running this same smoke-test wiring check will get the same behavior. There is no "dry run" / "smoke test" mode for the daemon. The Step 2 spec asked the operator to "manually populate the training database with enough transitions to cross min_transitions" — anyone who follows that literally will get exactly this cascade.

**Fix path (lower priority than F1/F2/F3):** add a daemon flag like `--no-train-on-start` or a config option `dry_run: true` that lets the daemon's check loop run but skips the actual trainer/promotion call. Alternative: document in `documentation/soak-test.md` Section 2.5 that the smoke test should be done with `min_transitions` raised to e.g. 1000000 to prevent the trigger from firing.

### Finding 5 — MAJOR — `taskkill /T` requires `cmd.exe` wrapper in Git Bash
**Category:** docs
**Evidence:** first kill attempt failed with `ERROR: Invalid argument/option - 'T:/'` — Git Bash converted `/T` to a path

Direct invocation of `taskkill /T /F /PID <pid>` from Git Bash mangles the `/T` flag because Git Bash's MSYS path conversion turns `/T` into `T:/`. The `feedback_buildstep_ui_kill_servers.md` memory mentions Windows server cleanup quirks, this is a related but distinct issue.

**Workaround that works:** `cmd.exe //c "taskkill /T /F /PID <pid>"` (the `//c` is the Git Bash escape for `/c` to cmd.exe).

**Fix path:** update `documentation/soak-test.md` Section 5.1 (clean stop) to document this exact incantation. Possibly also update `feedback_buildstep_ui_kill_servers.md` memory.
