## User hint (received mid-iteration)

> "The units are just moving and not attack walking, the computer was in a winning position and lost because it didn't just stand and fight."

**This reframes the entire run.** The primary bottleneck is not reward shaping — it's a command primitive: combat units getting `unit.move()` instead of `unit.attack()` when advancing, so they ignore enemies along the path and lose winning positions.

## Root cause located

[src/alpha4gate/bot.py:866-872](src/alpha4gate/bot.py#L866-L872):

```python
elif cmd.action == "move" and cmd.target_position is not None:
    ...
    if state in (StrategicState.ATTACK, StrategicState.LATE_GAME):
        unit.attack(Point2(cmd.target_position))
    else:
        unit.move(Point2(cmd.target_position))   # <-- runs past enemies
```

When strategic state is OPENING / DEFEND / EXPAND, a "move" action uses `unit.move()` — the unit ignores nearby enemies. During diff-5 games the bot shuffles between states frequently; any re-positioning during DEFEND/EXPAND becomes a free shot for the enemy.

**Fix (dev-type, needs `--self-improve-code`):**
- Replace the fall-through `unit.move()` with `unit.attack(target_position)` for combat units (non-worker).
- Or: add a pre-move engage check — if `unit.known_enemies_in_range` is non-empty, issue `attack()` on closest enemy; otherwise attack-move to target.
- Keep plain `move()` only for explicit non-combat operations (probe dispatch, warp-prism pickup, etc.) — the `scout.move(...)` calls on lines 787/807 are fine.

## Iteration decision

**No reward-rule or hyperparam change attempted this iteration.** A training change cannot fix a command-primitive bug — PPO choosing to "move here" still produces `unit.move()`. This must be fixed in code.
