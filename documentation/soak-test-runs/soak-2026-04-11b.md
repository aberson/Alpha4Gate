## Major finding — Phase 4.6 Step 1 regression breaks evaluator's game_id lookup (PRIMARY BLOCKER)

All 12 completed eval games were flagged as `crashed` by `evaluator._run_single_game`,
but the `games` table in `training.db` actually has **12 rows with real results** —
11 losses and **1 win** (`eval_v5_3c9bd047_d7b2ef1dde4e`, 619.8s, total_reward 0.157).
The result rows ARE being written correctly. The evaluator just can't find them.

### Root cause

- **`evaluator.py:104`** (unchanged by Phase 4.6) creates `game_id = f"eval_{checkpoint_name}_{uuid.uuid4().hex[:8]}"` — e.g. `eval_v5_3c9bd047`.
- That id is passed through `SC2Env.__init__(..., game_id=game_id)` → stored as `_base_game_id`.
- **`environment.py:172-173`** (**Phase 4.6 Step 1 commit `b27c6cc`**):

  ```python
  self._base_game_id = base
  self._game_id = f"{base}_{uuid.uuid4().hex[:12]}"
  ```

  Every `reset()` regenerates `_game_id` by appending a 12-char uuid hex suffix.
  This was the fix for the game_id collision finding from soak-3.
- Env writes the game row to the DB using `_game_id` — the SUFFIXED id
  `eval_v5_3c9bd047_d7b2ef1dde4e`.
- `evaluator.py:339` calls `self._get_game_result(game_id)` with the ORIGINAL
  (unsuffixed) id. `TrainingDB.get_game_result` does exact-match on the primary
  key → returns `None`.
- `evaluator.py:345-348` fires the `no result row was recorded` ERROR, flags the
  game as `crashed`, and refuses to count it toward win rate.

### Why Phase 4.6 didn't catch this

Phase 4.6 Step 1's gauntlet "PASS iter 2/3 (test quality flagged 4 redundant tests +
missing integration test in iter 1)". The integration test added in iter 2 exercised
the collision fix in isolation (multiple games in one env → unique ids) but did NOT
exercise the full **evaluator → env → db → evaluator-lookup** round trip. That round
trip is where the two sides of the id disagree.

Step 1's reviewer also didn't grep for every downstream caller of `get_game_result`
with the pre-fix id shape. One grep of `src/alpha4gate/` for `get_game_result` would
have turned up `evaluator._get_game_result` as a caller whose argument shape didn't
match the new env side.

### Secondary finding — env thread still hangs in 6 of 12 games (unrelated to the primary bug)

Even after fixing the game_id lookup, the `environment.py ERROR Timeout waiting for
observation from game thread` pattern fired in 6 of 12 eval games. These "flavor A"
crashes are real env-teardown bugs — the game thread doesn't exit cleanly after
sc2's `Status.ended` signal, and `environment.py`'s 5-minute observation timeout has
to fire to unblock the evaluator.

Under the current evaluator bug, these are invisible (all games look crashed
anyway). After the primary fix lands, these 5-minute stalls will become the dominant
performance problem in eval: **at ~5 min per flavor-A crash, 20 eval games ≈ 50
minutes minimum** regardless of bot skill. Needs a separate fix in the env teardown
path — likely sibling to #70 but in `environment.py:_obs_queue` consumer loop.

### Observability gap — #73 watchdog doesn't cover eval phase

`error_count_since_start` reached 18 during the soak; `daemon.last_error` stayed `null`
throughout. The #73 watchdog is scoped to cycles in `orchestrator.run()` and does not
poll during eval. Eval-phase crashes are silent to the dashboard's daemon-health
surface. Needs a sibling watchdog or a scope extension for Phase 4.7.

### Recommendation for Phase 4.7 build-phase

1. **Step 1 (BLOCKER): Fix evaluator / env game_id disagreement.** Three reasonable
   options — pick one during plan review:
   - (a) Env exposes `game_id` read-only property; evaluator reads it after `reset()` and uses that for lookup.
   - (b) Env takes `regenerate_game_id: bool = True` kwarg; evaluator passes
     `False` since it already supplies a unique id per game.
   - (c) Trainer/evaluator both pass their own fully-unique ids; env drops the
     regeneration entirely (Step 1's collision protection moves into the caller).

   Add the integration test that would have caught this in Phase 4.6: real env +
   real db + evaluator's exact lookup path, asserting the written row equals the
   queried row. Regression test also asserts the `games.game_id` PK matches the
   id the evaluator used to look it up.
2. **Step 2: Extend #73 watchdog to eval phase.** Re-arm the watchdog thread
   during eval, or add a sibling poller in the evaluator. Either way,
   `daemon.last_error` should surface if eval crashes pile up past threshold.
3. **Step 3: Diagnose env teardown hang (#70 sibling in eval).** Flavor A stalls
   waste 5 minutes per crash. Investigate `environment.py` `_obs_queue` consumer
   loop — likely the queue never sees a terminal sentinel after sc2's
   `Status.ended`. Fix before enabling longer soaks.
4. **Step 4 (docs): soak-test.md §3.2 stale line-count gate.** Replace
   "line count > 50 within 30 seconds" with a substring check for
   "Uvicorn running" + "Training daemon started". Also document the correct
   line-count baseline for the current backend (~7 lines).
5. **Step 5 (docs): Trainer log "10 games, ~150 timesteps" is misleading.** One
   game per cycle under `realtime=False`. Rename the log line to talk about
   timesteps, not game count, so operators don't chase a phantom "where are the
   other 9 games" question.
