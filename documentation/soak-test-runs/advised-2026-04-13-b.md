## CRITICAL FINDING: Advisor -> Training Pipeline Disconnect

### The Problem
Observation/validation games use `--decision-mode rules` (default). This means the **PPO
neural policy is never consulted** during gameplay. The rule-based DecisionEngine controls
strategic state (ATTACK/DEFEND/EXPAND) independently. Reward rule changes shape PPO
training signals, but PPO only runs inside the daemon training environment — never in
regular games unless `--decision-mode hybrid --model-path <checkpoint>` is explicitly passed.

### Evidence
1. All 4 iterations showed identical behavior because games always use the same rule-based
   decision engine regardless of reward changes
2. A hybrid-mode test game showed PPO being overridden by DEFEND rule every time
   `enemy_army_near_base` was True (which is frequent at diff 3)
3. Training soak completed only 2/4 target games — checkpoint not updated with new rewards

### Architecture (as discovered)
```
Regular game:  runner -> Alpha4GateBot -> DecisionEngine (rules) -> state
                                          PPO model: IGNORED

Training:      daemon -> _FullTrainingBot -> PPO action -> state override
                                             Reward rules -> PPO gradient

Hybrid game:   runner -> Alpha4GateBot -> NeuralDecisionEngine -> state
                                          BUT: enemy near base -> DEFEND override
```

### Specific Mismatches
1. **Bot class differs**: Training uses `_FullTrainingBot` (PPO controls state);
   regular games use `Alpha4GateBot` (DecisionEngine controls state)
2. **Observation vector differs**: Training includes advisor bridge features;
   regular games do not (even with Claude advisor active, it's a different codepath)
3. **Hybrid DEFEND override too aggressive**: Fires whenever `enemy_army_near_base` is True,
   which prevents PPO from ever choosing ATTACK when it matters most
