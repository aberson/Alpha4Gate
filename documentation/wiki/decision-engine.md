# Decision Engine

The strategic state machine that decides what the bot should be doing.

> **At a glance:** Six states (OPENING, EXPAND, ATTACK, DEFEND, LATE_GAME, FORTIFY)
> with priority-based transitions. DEFEND always wins if enemy near base. Command
> overrides from human/AI temporarily force a state. Thresholds are tunable constants.
> No SC2 imports — operates on the generic GameSnapshot dataclass.

> **Framework position:** internals of THE TASK. Runs inside each game; its action space (6 strategic states) is what the PPO policy predicts during training. See [training-pipeline.md](training-pipeline.md) for how states map to neural action indices.

## Purpose & Design

The decision engine evaluates the current game state every step and returns a
`StrategicState` enum. All downstream systems (macro, micro, fortification, commands)
key their behavior off this state. The neural engine can override the state machine's
output when in NEURAL or HYBRID mode.

### State transitions

```
              ┌──────────────────────────────────────────────┐
              │              COMMAND OVERRIDE                  │
              │  (highest priority, expires after duration)    │
              └──────────────────┬─────────────────────────────┘
                                 │ if no override
              ┌──────────────────▼─────────────────────────────┐
              │  enemy_army_near_base? ──yes──> DEFEND          │
              │         │ no                                    │
              │  recently_retreated AND outgunned? ──yes──> FORTIFY
              │         │ no                                    │
              │  build_order_incomplete? ──yes──> OPENING       │
              │         │ no                                    │
              │  3+ bases AND 480+ seconds? ──yes──> LATE_GAME  │
              │         │ no                                    │
              │  army_supply >= 12? ──yes──> ATTACK             │
              │         │ no                                    │
              │         └──> EXPAND                             │
              └────────────────────────────────────────────────┘
```

**Counterattack:** When leaving DEFEND (enemy retreated) and
`army_supply >= ATTACK_ARMY_SUPPLY`, transitions to ATTACK instead of EXPAND.

**Late-game attack:** LATE_GAME runs attack micro (attack rally + combat commands),
not just idle rally.

FORTIFY exits when: enemy supply drops to 0 OR own supply reaches
`enemy_supply * attack_supply_ratio`.

### Key thresholds

| Constant | Value | Purpose |
|----------|-------|---------|
| `ATTACK_ARMY_SUPPLY` | 12 | Minimum army supply to transition to ATTACK |
| `LATE_GAME_BASE_COUNT` | 3 | Bases needed for LATE_GAME |
| `LATE_GAME_TIME_SECONDS` | 480.0 | Time (8 min) needed for LATE_GAME |
| `fortify_trigger_ratio` | 1.5 (default) | Enemy/own supply ratio to trigger FORTIFY |
| `attack_supply_ratio` | 1.2 (default) | Own/enemy ratio to exit FORTIFY |

---

## Key Interfaces

`DecisionEngine.evaluate(snapshot: GameSnapshot, game_step: int) → StrategicState`

Also:
- `set_command_override(state, source, duration=120.0)` — force state from commands
- `notify_retreat()` — sets recently_retreated flag (feeds FORTIFY trigger)
- `set_claude_advice(advice)` — stores for decision log entries
- `decision_log` property — list of `DecisionEntry` (from_state, to_state, reason, claude_advice)

---

## Implementation Notes

`GameSnapshot` is a dataclass with 17 fields (same fields as the feature vector).
The state machine uses only: `army_supply`, `base_count`, `game_time_seconds`,
`enemy_army_near_base`, `enemy_army_supply_visible`.

Decision log entries include human-readable reasons generated by `_transition_reason()`.
